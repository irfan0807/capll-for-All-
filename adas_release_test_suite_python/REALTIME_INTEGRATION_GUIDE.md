# ADAS Test Suite — Mock Data and Real-Time Integration Guide

**Version:** 1.0  
**Date:** 23 April 2026  
**Scope:** Explains where mock data lives, how the suite executes in mock mode, and how to connect to a real-time ADAS software stack for release validation.

---

## 1) Purpose of This Document

This guide clarifies three practical questions:

1. Where mock data comes from in this test suite.
2. How the suite executes in real-time mode against a live software endpoint.
3. Which software components are typically used in automotive projects and how they are connected.

The goal is to help teams move from local deterministic testing (`mock`) to real release-gate execution (`http`) without changing test logic.

---

## 2) Current Architecture in This Repository

### 2.1 Main Components

- `test_catalog.py`
  - Builds the 250 test catalog (50 smoke + 200 regression).
  - Contains scenario generation (input values) for AEB, FCW, ACC, LKA, BSD, TSR, DMS.
  - Contains `_oracle(...)`, which produces expected signals for each scenario in mock mode.

- `sut_adapter.py`
  - Runtime switch between `mock` and `http` SUT modes.
  - `mock`: expected behavior generated internally by `_oracle(...)`.
  - `http`: sends request to external ADAS software endpoint.

- `assertions.py`
  - Validates output contract and expected signals.
  - Supports strict signal mode (`ADAS_STRICT_SIGNALS=1`) to reject extra keys.

- `run_release_gate.py`
  - Release gate runner (smoke/regression/all).
  - Produces versioned artifacts (`junit.xml`, `report.html`, `summary.json`).
  - Applies production thresholds (`min-pass-rate`, `max-failures`, `max-errors`).

- `ui_app.py`
  - Human interaction UI (Streamlit) for running suites and reviewing reports.

## 3) Where Mock Data Is and How It Works

### 3.1 Mock Input Data

Mock input data is generated in `test_catalog.py` by feature-specific generators:

- `_generate_aeb()`
- `_generate_fcw()`
- `_generate_acc()`
- `_generate_lka()`
- `_generate_bsd()`
- `_generate_tsr()`
- `_generate_dms()`

Each generator creates scenario dictionaries with boundary values and edge cases. Example fields include:

- `ego_speed_mps`
- `target_distance_m`
- `closing_speed_mps`
- `road_friction`
- `visibility_m`
- `lane_confidence`
- `eyes_off_road_s`
- `sensor_ok`

### 3.2 Mock Expected Data

Expected responses are generated by `_oracle(feature, scenario)` in `test_catalog.py`.

The oracle returns a normalized output shape:

```json
{
  "status": "OK",
  "signals": {
    "aeb_warn": true,
    "aeb_brake": false
  }
}
```

If sensor health is invalid (`sensor_ok=false`), oracle returns failsafe output:

```json
{
  "status": "FAILSAFE",
  "signals": {
    "failsafe": true
  }
}
```

### 3.3 Why This Design Is Useful

- Deterministic: no infrastructure dependency during development.
- Fast: 250-case suite executes in seconds.
- Stable release baseline: supports CI gate reliability.
- Same test cases can run in `http` mode without rewriting tests.

## 4) Real-Time Mode: How External Connection Works

### 4.1 Mode Switching

`ADASSUTAdapter` in `sut_adapter.py` uses:

- `ADAS_SUT_MODE=mock` (default)
- `ADAS_SUT_MODE=http` (real service)

When in `http`, endpoint URL must be configured:

- `ADAS_SUT_URL=http://<host>:<port>/predict`

### 4.2 HTTP Request/Response Contract

#### Request

```json
{
  "feature": "AEB",
  "scenario": {
    "ego_speed_mps": 22.2,
    "target_distance_m": 10.0,
    "closing_speed_mps": 8.0,
    "road_friction": 0.8,
    "sensor_ok": true
  }
}
```

#### Response (minimum contract)

```json
{
  "status": "OK",
  "signals": {
    "aeb_warn": true,
    "aeb_brake": false
  }
}
```

Contract requirements enforced by test suite:

- JSON object root
- `status` key exists and is string
- `signals` key exists and is object
- Required expected signal keys must be present
- Expected value types and values must match
- Optional strict mode can fail on unexpected extra signal keys

### 4.3 Retry/Timeout Behavior

Adapter supports production resilience:

- Configurable timeout: `ADAS_SUT_TIMEOUT_S`
- Configurable retries: `ADAS_SUT_HTTP_RETRIES`
- Configurable backoff: `ADAS_SUT_BACKOFF_S`
- Retries on transient HTTP statuses (429/5xx)

## 5) Typical Automotive Real-Time Software Options

The suite is transport-agnostic at test level (it only needs HTTP contract), so you can connect to multiple backends.

### Option A: Python FastAPI Service (rapid prototyping)

Use when:

- Validating model logic quickly
- Building CI-friendly software-in-the-loop APIs
- Early release-gate setup

Pros:

- Fast implementation
- Easy JSON handling
- Good integration with ML inference stacks

### Option B: C++ Service (high-performance production path)

Use when:

- Production inference/runtime is C++
- Lower latency requirements
- Better alignment with embedded gateway codebase

Pros:

- Performance and memory control
- Closer parity with production components

### Option C: HIL/Middleware Gateway API

Use when:

- Need to validate against real ECU/HIL signals
- Vehicle network data is available from CANoe/HIL benches

Typical pattern:

- Gateway subscribes to bus/simulation signals
- Gateway maps current signal frame into `scenario`
- Gateway returns normalized `status/signals`

## 6) End-to-End Connectivity Patterns

### 6.1 Software-in-the-Loop (SIL) Path

1. Test case loaded from catalog.
2. Adapter posts to `/predict`.
3. Model service computes ADAS response.
4. Assertion compares response to expected.
5. Gate runner aggregates pass/fail metrics.

Best for:

- Release gating in CI
- Fast nightly regression
- Algorithm validation against API contract

### 6.2 HIL-Backed API Path

1. Test case posted to gateway service.
2. Gateway injects conditions to HIL environment (or maps to simulation control).
3. HIL system executes ECU logic.
4. Gateway collects output signals and returns normalized response.
5. Assertions validate result and gate records artifact.

Best for:

- Hardware + integration confidence before release
- End-to-end signal-chain verification

## 7) Practical Setup Steps for Real-Time Execution

### 7.1 Environment Configuration

Set mode and endpoint:

- `export ADAS_SUT_MODE=http`
- `export ADAS_SUT_URL=http://<host>:<port>/predict`

Optional hardening:

- `export ADAS_SUT_TIMEOUT_S=8`
- `export ADAS_SUT_HTTP_RETRIES=2`
- `export ADAS_SUT_BACKOFF_S=0.2`
- `export ADAS_STRICT_SIGNALS=1`

### 7.2 Run via Release Gate

Examples:

- Smoke:
  - `python run_release_gate.py --suite smoke --build-id adas-2026.04.23-rc1`

- Regression:
  - `python run_release_gate.py --suite regression --build-id adas-2026.04.23-rc1`

- Full:
  - `python run_release_gate.py --suite all --build-id adas-2026.04.23-rc1`

### 7.3 Artifacts to Review

For each run:

- `junit.xml`: machine-readable test results
- `report.html`: human-readable execution report
- `summary.json`: gate status, thresholds, pass rate, metadata

## 8) Data Mapping Guidance for Real ADAS Teams

When connecting to real systems, keep this normalization layer stable:

- Input mapping layer: Convert platform-specific signal names/units into canonical `scenario` keys.
- Output mapping layer: Convert model/ECU-specific output into canonical `signals` keys expected by test suite.

Recommended controls:

- Unit normalization (kph ↔ mps, meters, seconds)
- Explicit handling of unavailable sensors
- Failsafe output normalization (`status=FAILSAFE`)
- Versioned schema for backward compatibility

## 9) Recommended Production Deployment Pattern

### 9.1 CI/CD

- Trigger release gate per build artifact (manual or automatic).
- Use strict thresholds for release branches:
  - `min-pass-rate=100`
  - `max-failures=0`
  - `max-errors=0`

### 9.2 Execution Tiers

- Tier 1: smoke on every candidate build
- Tier 2: full regression nightly
- Tier 3: selected HIL-backed suite before release freeze

### 9.3 Traceability

Store with build ID:

- Test report artifacts
- Service version/hash
- Model version/hash
- API schema version

This makes release decisions auditable.

---

## 10) Common Integration Risks and Mitigation

1. **Schema drift** (service response changes)
   - Mitigation: strict contract assertions + schema versioning.

2. **Unit mismatch** (kph vs mps)
   - Mitigation: explicit mapping layer and conversion tests.

3. **Intermittent service failures**
   - Mitigation: retry/backoff/timeouts and health checks.

4. **Non-deterministic model output**
   - Mitigation: tolerance windows where needed and deterministic inference mode for gate runs.

5. **Signal key expansion breaks strict checks**
   - Mitigation: run with strict mode on release branches, relaxed mode in exploratory branches.

---

## 11) Summary

- Mock data and expected behavior are in `test_catalog.py`.
- Runtime switching is in `sut_adapter.py` (`mock` vs `http`).
- Real-time integration requires an HTTP `/predict` endpoint with `status/signals` contract.
- The gate runner (`run_release_gate.py`) already supports production thresholds and artifact traceability.
- The same 250-case catalog can be executed from local mock to real service/HIL-backed API with no test rewrite.

This architecture is intentionally designed for gradual maturity: **start deterministic, then connect real software, then connect real benches**.
